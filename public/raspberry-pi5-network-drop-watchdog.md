## 症状

Raspberry Pi 5（Wi-Fi 運用、ヘッドレス）を SSH と Tailscale で常用しているのですが、ときどき **電源は入っている（LED は点灯）のに、ネットワークから完全に消える** ことがありました。

- LAN の SSH → timeout
- Tailscale の SSH → timeout
- RDP も timeout

しかも繰り返し起きて、毎回「電源を入れ直す」で復旧させていました。これを **原因の切り分け** と **自動復旧** までやったので残します。

## 切り分け1：`tailscale status` で「機器側か自分側か」を一発判別

SSH が通らないとき、つい「ラズパイが落ちた」と決めつけがちですが、**自分側（PC の Wi-Fi / VPN）が原因のこともある**ので、まずそこを切り分けます。

Tailscale を使っているなら `tailscale status` が一番速いです。

```text
100.x.y.z   raspberrypi   user@   linux   active; relay "..."; offline, last seen 1h ago, tx 124956 rx 0
```

ポイントは peer 行の **`offline, last seen 1h ago`**。これは Tailscale のコーディネーションサーバが持っている記録で、**自分の SSH とは独立した第三者の事実**です。

- peer が `offline` → 本当にラズパイ側がネットから落ちている
- peer が `active`/オンライン → 落ちているのは自分側（PC の VPN/ネット）

さらに `last seen` で **いつ落ちたか** も分かります。`tx ... rx 0` は「こちらからは送れているが相手から1バイトも返ってこない」状態です。

自分の PC 側も `tailscale status --json` の `BackendState=Running` / `Self.Online=true` で「自分は正常」を確認しておきます。

## 切り分け2：復旧後に `journalctl -b -1`（**前の** boot）を読む

ここが一番ハマりどころです。ネットから消えている間は当然ログを取りに行けません。復旧（再起動）してから読むことになりますが、**再起動した瞬間に「現在の boot」は新しくなる**ので、`journalctl`（デフォルト）や `dmesg` を見ても落ちた瞬間が入っていません。

落ちたときのログは **1つ前の boot** にあります。

```bash
# 直前の boot のログ全体（末尾）
journalctl -b -1 --no-pager | tail -30

# 落ちた当日の wlan0 / ネットワーク関連だけ
journalctl -b -1 --no-pager | grep -iE "wlan0|cfg80211|brcmfmac|LinkChange|network is unreachable" | tail -40
```

自分のケースで出ていたもの（時刻順）：

```text
19:57  avahi-daemon: Withdrawing address record ... on wlan0   # IPv6 アドレスが flap
20:08  tailscaled: LinkChange: major, rebinding ... rebind-reason=[time-jumped(13m50s),ips-changed,protocols-changed]
22:06  tailscaled: ... connect: network is unreachable          # 完全に不通
```

重要なのは、**この間もログは出続けていた**こと。つまり OS 全体のフリーズや OOM ではなく、**wlan0／ネットワーク層だけが落ちて不通になっていた**わけです（`journalctl -b -1` の末尾に `oom` も `panic` もなし）。

> 根本トリガー（なぜ wlan0 が落ちたか）まではログから特定しきれませんでした。ドライバの明示的な crash 行も出ておらず、`vcgencmd get_throttled` も復旧後は `0x0`（電圧低下フラグなし）。なので以降は「**wlan0 が落ちる事象一般への対症**」として watchdog を入れます。

## 対策：NetworkManager を再起動する watchdog

「OS は動いているのにネットだけ落ちる」なら、**cron で監視して、落ちていたらネットワークを再起動すれば自動復旧できる**はずです。Pi OS は最近 NetworkManager 運用（`resolv.conf` が `# Generated by NetworkManager`）なので、それを再起動します。

`/home/pi/.local/bin/net-watchdog.sh`：

```bash
#!/usr/bin/env bash
# gateway が不達なら NetworkManager を再起動して wlan0 を復旧させる
GATEWAY=192.168.1.1   # 自宅ルータのIPに置き換え
LOG=/home/pi/logs/net-watchdog.log
if ping -c 3 -W 3 "$GATEWAY" >/dev/null 2>&1; then
  exit 0
fi
echo "$(date -Is) gateway $GATEWAY unreachable, restarting NetworkManager" >> "$LOG"
systemctl restart NetworkManager
```

`/etc/cron.d/net-watchdog`（3分ごと、root 実行）：

```text
*/3 * * * * root /home/pi/.local/bin/net-watchdog.sh
```

インストール：

```bash
chmod +x /home/pi/.local/bin/net-watchdog.sh
mkdir -p /home/pi/logs
echo "*/3 * * * * root /home/pi/.local/bin/net-watchdog.sh" | sudo tee /etc/cron.d/net-watchdog
```

### 誤再起動しないことを確認

ネットが正常なときに余計な再起動をされると困るので、`ping` が通る状態で1回手で走らせて確認します。

```bash
/home/pi/.local/bin/net-watchdog.sh
echo $?        # → 0
cat /home/pi/logs/net-watchdog.log   # → ファイルなし（=再起動していない）
```

`ping` が通る間はログにも何も残らず、`systemctl restart` も走りません。落ちたときだけ動きます。

## まとめ

- 「電源 ON なのにネットから消える」は **ネットワーク層のドロップ**であることが多く、OS は生きている。
- 切り分けは **`tailscale status`（第三者の offline 記録）→ 復旧後に `journalctl -b -1`（前の boot）** の順が速い。
- 対症療法として **gateway を ping して落ちていたら NetworkManager を再起動する watchdog** を cron で入れておくと、OS が動いている限り数分で自動復旧する。
- ただし OS 自体が止まる系の障害には cron 駆動の watchdog は効かない点は理解しておく。

根本原因の特定はまだですが、少なくとも「毎回手で電源を入れ直す」運用からは抜けられました。
