## 結論から：使うべきか？

**YES。ただし使用量制限を意識して戦略的に使うべき**

Claude Pro（月額20ドル）には**追加料金なしで**Claude Codeへのアクセスが含まれますが、claude.ai（Web版）と使用量を共有します。5時間あたり約10〜40回のプロンプトという制限があるため、**「コード品質の最終チェック」として使うのが最適**です。

---

## 📦 初回セットアップ（5分で完了）

### 1. Claude Codeのインストール

```bash
# macOS/Linux
brew install anthropics/tap/claude-code

# または公式インストーラー
curl -fsSL https://cli.claude.ai/install.sh | sh
```

### 2. 認証

```bash
claude login
```

ブラウザが開き、Claude Proアカウントでログイン。これで準備完了です。

### 3. code-simplifierのインストール

**方法1：CLIから直接（推奨・簡単）**

プロジェクトディレクトリで：
```bash
claude plugin install code-simplifier
```

**方法2：セッション内でインストール**

```bash
claude  # セッション開始

# セッション内で
/plugin marketplace add anthropics/claude-plugins-official
/plugin install code-simplifier
```

## 🎯 Claude Proユーザーが使うべき具体的なタイミング

### ✅ 使うべきとき

| タイミング | 理由 | 例 |
|----------|------|-----|
| **PR提出前** | レビュアーの負担を減らす | 「このPRの変更を整理して」 |
| **長時間コーディング後** | 試行錯誤の痕跡を消す | 「今日変更したファイルをクリーンアップ」 |
| **レガシーコード修正後** | 古いコードと新しいコードの統一 | 「この関数を現代的なスタイルに」 |
| **複雑な関数を書いた直後** | 自分でも読みにくいと感じたら | 「このロジックをシンプルにして」 |

### ❌ 使わなくていいとき

- 最初から綺麗に書けている小さな変更
- 既に十分シンプルなコード
- 公開API変更が必要な大規模リファクタリング（人間が判断すべき）
- 単純な質問や調査（通常の `claude` コマンドで十分）

## 💡 実践的な使い方（コピペで使える例）

### シナリオ1：PR提出前のクリーンアップ

```bash
# プロジェクトディレクトリで
claude

# セッション開始後
> このブランチの変更をレビューしやすくして。
> 機能は変えず、コードの読みやすさだけ改善して。
> 変更点をまとめて教えて。
```

**想定される作業時間：** 3〜5分  
**消費プロンプト数：** 2〜3回

---

### シナリオ2：複雑な関数のリファクタリング

```bash
claude

> src/utils/validation.jsのvalidateUserInput関数を見て。
> ネストが深すぎて読みにくいから、early returnパターンで書き直して。
> 機能は完全に同じままで。
```

**想定される作業時間：** 2〜3分  
**消費プロンプト数：** 1〜2回

---

### シナリオ3：長時間セッション後の一括整理

```bash
claude

> 過去2時間で変更したファイルをすべて確認して、以下を修正：
> 1. デバッグ用のconsole.logを削除
> 2. 不要なコメントを削除
> 3. 変数名を統一
> 4. テストを実行して動作確認
```

**想定される作業時間：** 5〜10分  
**消費プロンプト数：** 3〜5回

---

### シナリオ4：プロジェクト規約に沿った整理

**まず `.claude/CLAUDE.md` を作成：**

```bash
mkdir -p .claude
cat > .claude/CLAUDE.md << 'EOF'
# コーディング規約

- 関数名: キャメルケース（getUserData）
- 定数: スネークケース大文字（MAX_RETRY_COUNT）
- インデント: スペース2つ
- 文字列: シングルクォート推奨
- 不要なコメントは削除
EOF
```

**その後：**
```bash
claude

> このディレクトリ全体をCLAUDE.mdの規約に沿って整理して
```

code-simplifierは自動的に `.claude/CLAUDE.md` を読み込み、プロジェクト固有のルールに従います。

## ⚙️ 使用量を節約する賢い使い方

### 戦略1：ファイルを絞り込む

❌ 非効率：
```
> プロジェクト全体を整理して
```

✅ 効率的：
```
> src/auth.js と src/validation.js だけシンプルにして
```

### 戦略2：段階的に指示

❌ 非効率（やり直しが発生）：
```
> このコードをシンプルにして
[結果を見て] あ、これじゃない...
> もっと関数を小さくして
[再度処理]
```

✅ 効率的：
```
> このコードをシンプルにして。具体的には：
> - 関数を50行以下に分割
> - ネストは3段階まで
> - 変数名を明確に
```

### 戦略3：Web版Claudeと使い分ける

| タスク | 使うツール | 理由 |
|-------|----------|------|
| コード設計の相談 | Web版Claude | ファイル操作不要 |
| エラー調査 | Web版Claude | コピペで十分 |
| 実装＋整理 | Claude Code | ファイル直接編集が必要 |
| 複数ファイルの一括整理 | Claude Code + code-simplifier | 自動化の恩恵大 |

## 🚀 推奨ワークフロー

### デイリー開発での使い方

```
【朝】プロジェクト開始
├─ Web版Claude: 今日のタスク設計

【日中】実装
├─ 自分で実装
├─ 詰まったらWeb版Claudeで相談

【夕方】コード品質チェック
├─ Claude Code起動
├─ code-simplifierで整理
├─ テスト実行
└─ コミット・PR作成
```

### PR作成時の使い方

```bash
# 1. 変更をコミット（整理前）
git add .
git commit -m "feat: 新機能実装（整理前）"

# 2. Claude Codeで整理
claude
> このブランチの変更をレビューしやすく整理して

# 3. 差分確認
git diff

# 4. テスト実行
> テストを実行して

# 5. 問題なければコミット修正
git add .
git commit --amend -m "feat: 新機能実装"
```

## 📊 使用量モニタリング

### 残り使用量の確認

```bash
claude usage
```

### 上限に達した場合の対処

Claude Proは5時間ごとにリセットされます：

- **緊急性が高い** → 5時間待つ
- **本当に緊急** → MaxプランにアップグレードまたはAPI使用を検討
- **予防策** → Web版Claudeと使い分けを徹底

```bash
# プラグインリストを確認
/plugin list

# 再インストール
/plugin uninstall code-simplifier
/plugin install code-simplifier
```

### Q2: 変更が期待と違う

```
> この変更は違う。元に戻して
> 代わりに〜〜してほしい
```

Git管理していれば `git checkout -- <ファイル>` でも復元可能。

### Q3: 使用量がすぐ上限に達する

- 1プロンプトで複数のタスクを依頼
- ファイルを絞り込む
- 単純な質問はWeb版Claudeへ

## まとめ：Claude Proユーザーの最適な使い方

✅ **追加料金なしで使える**  
✅ **PR提出前・長時間作業後の「最終仕上げ」として最適**  
✅ **使用量制限があるため、Web版Claudeと戦略的に使い分ける**  
✅ **`.claude/CLAUDE.md` でプロジェクト規約を定義すると効果倍増**  

code-simplifierは「コードレビュー前の自動整形ツール」として捉え、日々の実装は自分で行い、仕上げをClaude Codeに任せるのが、Claude Proプランでの賢い使い方です。
